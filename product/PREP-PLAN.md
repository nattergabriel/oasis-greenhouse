# Pre-Hackathon Prep Plan

> What to prepare before and during the hackathon, in dependency order. Each layer feeds into the next.

---

## Layer 0 — Case Partner & Market Understanding _(the foundation of everything)_

**Folder:** `research/`

- [ ] Read and annotate the challenge brief → `research/CASE-BRIEF.md`
- [ ] Research Syngenta deeply → `research/PARTNER-ANALYSIS.md`
  - Their products, digital landscape, customers, pain points
  - Their language (use "grower" not "user", etc.)
- [ ] Define market fit → `research/MARKET-FIT.md`
  - Who's the target user, what's the value prop, what's outside-the-box
- [ ] Build the business case → `research/BUSINESS-CASE.md`
  - Costs of the problem, value of the solution, monetization model
- [ ] Draft value roadmap → `research/VALUE-ROADMAP.md`
  - MVP → pilot → scale → mature (shows judges we think beyond the weekend)

**Why first:** Everything — the brand, the design, the pitch, the features — should be shaped by the case partner's world, not our assumptions. If we nail this, the pitch writes itself.

---

## Layer 1 — Brand Identity _(shaped by the case partner context)_

**File:** `brand/IDENTITY.md`

- [ ] Pick colors that feel appropriate for the partner's industry (agriculture = earthy? tech-forward? both?)
- [ ] Choose font pairing (heading + body)
- [ ] Decide on tone/voice that resonates with the case partner and their users
- [ ] Prepare logo creation workflow (name comes from ideation)
- [ ] Set up `brand/assets/` folder

**Why second:** Brand choices should reflect the problem space. An ag-tech solution for Syngenta feels different from a fintech app. Layer 0 informs these decisions.

---

## Layer 2 — Design System _(builds on brand)_

**File:** `design-system/SYSTEM.md`

- [ ] Agree on UI framework/library with frontend dev
- [ ] Define spacing, radius, shadows
- [ ] Document component patterns (buttons, cards, inputs)
- [ ] Decide: dark mode yes/no, responsive priority
- [ ] Export CSS variables / Tailwind config for frontend

**Why third:** Translates brand into code-ready decisions. Frontend dev can configure their project immediately.

---

## Layer 3 — Templates & Scaffolding _(builds on brand + design system)_

- [ ] Pitch deck skeleton with brand styling → `pitch/`
- [ ] Demo script outline → `pitch/DEMO-SCRIPT.md`
- [ ] Submission text template → `pitch/SUBMISSION.md`
- [ ] Website/app wireframes → `wireframes/`

**Why fourth:** Empty containers with the right look. Fill in content during crunch time instead of building from scratch.

---

## Layer 4 — Process & Checklists _(helps along the way)_

- [ ] Demo-day checklist (rehearsal, backup video, tech check)
- [ ] Submission checklist (README, description, links, screenshots)
- [ ] Standup template (2-hourly check-ins)

---

## Layer 5 — Reusable for the Future _(long-term value)_

- [ ] Keep brand kit adaptable
- [ ] Post-hackathon retro notes
- [ ] Template the pitch structure for reuse
- [ ] Clean archive of final assets

---
