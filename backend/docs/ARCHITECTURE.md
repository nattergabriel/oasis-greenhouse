# Backend Architecture

## Stack

- **Python + FastAPI** -- orchestrator, API for frontend
- **LangGraph** -- agent state machine
- **AWS Bedrock** -- LLM calls (via boto3)
- **MCP Knowledge Base** -- Syngenta-provided crop/environmental data
- **Separate sim engine server** -- stateless, built by teammate

## Two Modes

**Training mode**: Run simulations to improve the strategy document. No frontend involved. Fast loop.

**Demo mode**: Frontend loads a pre-computed simulation result and plays it back on a timeline. No SSE/streaming.

## API

```
POST /api/training/run              Run simulation, store result, improve strategy doc
     body: { config: {...}, inject_events: [{day: 120, type: "temperature_failure", severity: 0.7}] }
GET  /api/simulations               List past simulation runs
GET  /api/simulations/{id}          Get full result (450 days of data)
GET  /api/strategy                  Current strategy document
```

`inject_events` are pre-defined events applied at specific days during the simulation. These are in addition to random events generated by the sim engine.

## LangGraph State Machine

```
init -> plan -> simulate -> plan (batch done, days remaining)
                         -> react (event fired) -> simulate
                         -> reflect (day 450) -> done
```

### Nodes

**`init`** -- Calls sim engine `POST /simulate/init`. Loads strategy doc. Loads cached KB data.

**`plan`** -- Sends current state + strategy doc + cached crop profiles + nutrition targets to LLM. Gets back zone plans (percentage-based crop allocations), environment settings, and water allocations for the next ~30 days. The agent sets strategy; the sim engine handles planting, harvesting, and replanting automatically per zone plans.

**`simulate`** -- Sends state + actions to sim engine `POST /simulate/tick`. Crops are auto-harvested by the sim engine when ready. If sim returns early (event fired or threshold breached), routes to `react`. Otherwise routes back to `plan` or to `reflect` if mission complete.

**`react`** -- Like `plan` but with event context. Queries KB live for relevant stress/scenario guidance. Gets back response actions. Routes back to `simulate`.

**`reflect`** -- Post-simulation. Sends strategy doc + run metrics + key events to LLM. LLM rewrites strategy doc. Saves it.

### Routing Logic

```python
def route_after_simulate(state):
    if state.greenhouse.mission_day >= 450:
        return "reflect"
    if state.sim_result.stopped_early:
        return "react"  # event fired or threshold breached
    return "plan"
```

## KB Integration

**Cached at startup** (reused across all runs):
- Crop profiles (doc 03): growth cycles, yields, nutritional values, environmental thresholds
- Nutritional targets (doc 05): daily requirements for 4 astronauts

**Live queries during `react` node**:
- Stress response guides (doc 04): symptoms, mitigations per stress type
- Operational scenarios (doc 06): failure mode response chains

The agent also has a `query_knowledge_base` tool for ad-hoc KB lookups during planning.

**MCP endpoint**: `https://kb-start-hack-gateway-buyjtibfpg.gateway.bedrock-agentcore.us-east-2.amazonaws.com/mcp`
**Tool name**: `kb-start-hack-target___knowledge_base_retrieve`

## Learning System

Two knowledge sources injected into every LLM call:
- **KB data** (static facts): "Beans need 18-25C"
- **Strategy document** (evolves): "Start beans by day 5, stagger every 30 days"

After each simulation run, the LLM rewrites the strategy doc based on outcomes. It keeps what worked, drops what didn't. The doc stays concise (2-3 pages), never grows unboundedly.
